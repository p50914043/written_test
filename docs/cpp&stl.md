## 多态

### C++ 有两种多态：编译时多态和运行时多态，你能讲讲两种多态吗？

静态多态：
定义：编译器在编译阶段就确定了要调用的具体函数，也叫静态多态。
实现方式：主要通过 函数重载 和 模板 实现
特点：效率高
动态多态：
定义：编译器在编译阶段无法确定调用哪个函数，必须等到程序运行时，根据对象的实际类型才能确定，也叫动态多态。
实现方式：必须满足两个条件：
基类中声明虚函数（用virtual关键字）；
通过基类指针 / 引用指向派生类对象。

### 虚表的产生以及虚函数的访问过程

虚表是编译阶段生成的一个全局只读的函数指针数组，如果是派生类，如果重写了基类的虚函数，派生类虚表对应位置会替换为自身的函数指针 （虚表中还有类型信息，需要类中有虚函数且编译器启用了RTTI）

虚表指针（vptr) 是每个含虚函数的对象隐含的成员变量（通常在对象的最开头），用来指向所属的需函数表。（**vptr 赋值** ：在**构造函数的初始化列表执行完毕后，构造函数体执行之前）**

调用虚函数的访问流程： 取vptr -> 找vtable -> 根据偏移量找函数地址 -> 执行函数

### 多态基类声明虚构函数

基类指针指向派生类对象时，若析构函数非虚，会导致派生类析构函数不执行，引发内存泄露

### 类型强转static_cast、dynamic_cast\const_cast\reinterpret_cast

static_cast：是编译期完成的静态类型转换，无运行时校验、速度快，适用于基本类型转换、多态 / 非多态类的向上继承转换，但向下转换无安全保障，可能引发未定义行为；
dynamic_cast：仅用于多态类（含虚函数），在运行期依赖 RTTI 检查对象实际类型，转换失败时指针返回nullptr、引用抛异常，安全但有轻微性能开销，专用于多态类的安全向下 / 交叉继承转换；
const_cast：仅用于增减变量的 const/volatile 属性（不改变类型本身），使用不当易触发未定义行为
reinterpret_cast：是编译期直接重解释数据二进制位的强制转换，完全无视类型兼容性，风险极高，仅适用于底层 / 平台相关的特殊场景。

### labmda表达式

捕获方式：**值捕获、引用捕获、this捕获**
是否可修改捕获的变量：

* 捕获变量：捕获变量的初始值，即变量的当前值，在lambda函数内部无法修改捕获变量的值（默认时const属性）。当需要修改时，添加mutable，可修改捕获变量的值。如
  auto lambda = [[num]()] () mutable {
  num = 20; // 此时修改的是lambda内部的num拷贝，而非外部原变量
  std::cout << "lambda内部修改后的num：" << num << std::endl; // 输出：20
  };
* 引用捕获可以修改捕获变量的值

### placement new

普通的new运算符做两件事：

* 分配一块内存
* 在这块内存上调用对象的构造函数
  而placement new 则跳过内存分配，直接在指定的已分配内存地址上调用对象的构造函数

语法格式：
new (内存地址) 类型名(构造参数)

## 模板元编程TMP

核心是利用模板的编译期特性，让编译器在编译阶段完成计算、逻辑判断、类型推导等工作，而非运行

模板的应用场景总结：

* 标准库的类型萃取（type_traits）
  C++11 引入<type_traits>头文件（如std::is_pointer、std::is_array、std::enable_if）都是基于 TMP 实现的，用于编译期类型判断、类型转换。
* 编译器优化
  对固定的数值计算（如数学常量、固定长度的数组处理），用 TMP 在编译期计算完成，避免运行时开销（比如游戏引擎的数学库）。
* 静态断言（static_assert）
  结合 TMP 做编译期检查，比如 “确保模板参数是整数类型”：
  template `<typename T>`
  void func(T x) {
  static_assert(std::is_integral_v `<T>`, "T必须是整数类型！");
  // 业务逻辑
  }
* 策略模式（编译期选择算法）
  通过模板参数选择不同的算法实现，编译期确定调用逻辑，无运行时分支开销。
* 元组
  C++ 的能存储不同类型的元素，其底层的类型索引、元素访问都是基于 TMP 的递归实现。

### 智能指针

std::unique_ptr: 同一时间，仅一个unique_ptr能指向某块堆内存，独占所有权，禁止拷贝，但支持移动

shared_ptr：多个shared_ptr可指向同一堆内存，通过引用计数管理（强引用计数记录指向该资源的shared_ptr数量、弱引用计数指向该控制块的weak_ptr的数量

智能指针make_shared或make_unique的核心区别：

* 内存分配：直接构造会触发两次内存分配 直接构造会触发两次内存分配，比如

```
// 1. 直接构造 shared_ptr（手动 new）
std::shared_ptr<int> p1(new int(10)); // int的分配+shared_ptr控制块分配
// 2. 使用 std::make_shared（推荐写法）
std::shared_ptr<int> p2 = std::make_shared<int>(10); // 把「对象内存」和「控制块内存」合并在同一块连续的堆内存中
```

* 异常安全

直接构造存在内存泄漏风险，而make_shared能完全规避

```
// 危险！可能内存泄漏
void foo(std::shared_ptr<int> p, int value) {}

// 调用时：
foo(std::shared_ptr<int>(new int(10)), bar());  //C++ 函数参数的求值顺序是未定义的，假如new执行完后，bar()抛出异常
```

* make_shared\unique 更加便捷

但是有些场景必须用手动构造的方式：比如 需要自定义删除器、weak_ptr长期存活

weak_ptr： 配合std::shared_ptr使用的若引用智能指针，解决shared_ptr循环引用的问题。如果shared_ptr资源释放了，`weak_ptr`会自动无效。weak_ptr必须先通过lock()转shared_ptr后才能访问元素，lock()后必须判空

## 容器

### 容器以及对应的数据结构

| 容器          | 底层结构 | 有序性 | 随机访问 | 插入 / 删除（平均）    | 核心优势               |
| ------------- | -------- | ------ | -------- | ---------------------- | ---------------------- |
| vector        | 连续内存 | 无序   | O(1)     | 尾部 O (1)，中间 O (n) | 随机访问快、缓存友好   |
| deque         | 分段连续 | 无序   | O(1)     | 首尾 O (1)，中间 O (n) | 双端操作高效           |
| list          | 双向链表 | 无序   | O(n)     | 任意位置 O (1)         | 任意位置增删不移动元素 |
| map/set       | 红黑树   | 有序   | 无       | O(logn)                | 有序存储、范围查找     |
| unordered_map | 哈希表   | 无序   | 无       | O(1)                   | 按 key 快速查找        |

### vector扩容机制

添加元素且当前容量不足以容纳新元素时，就会触发扩容，扩容过程会迁移原数据到心内存（c++11前拷贝，c++11后改为移动（需要支持移动的类型）），释放旧内存并更新状态。

扩容倍数1.5 -2 倍

如何避免反复扩容：

* 提前reserve
* resize：适用于指定容量，且要提前初始化的情况
* 构造
* push_back

| 接口       | 作用             | 是否改变 size | 是否改变 capacity   | 适用场景         |
| ---------- | ---------------- | ------------- | ------------------- | ---------------- |
| reserve(n) | 预分配容量       | ❌            | ✅（仅增大）        | 避免频繁扩容     |
| resize(n)  | 调整实际元素个数 | ✅            | ✅（n>capacity 时） | 直接设置元素数量 |

### vector push_back和emplace_back的区别

push_back：先创建一个完整的对象，再将这个对象拷贝、移动到vector的内存中

emplace_back: 直接在vector的内存空间就地构造对象，跳过“先创建再拷贝/移动”的步骤

vector的底层实现原理

vector是动态数组，底层就是一段连续的内存，支持随机访问

包含三个关键指针：

T* _start; // 数组起始地址

T* _finish; // 最后一个有效元素的下一个位置

T* _end_of_storage; // 整个空间的末尾

size()\capacity则是通过这三个指针计算而来

## 迭代器

### 迭代器失效

容器因 **增删元素、内存扩容** ，导致原迭代器指向 **无效 / 错误内存** ，无法正常遍历、访问元素，即为迭代器失效。

场景：**vector/string 等连续内存容器**

* 插入触发 **扩容** ：所有迭代器直接失效；
* 中间插入 / 删除：插入 / 删除位置**之后**的迭代器失效。

2. **list 链表容器**
   仅**被删除元素**的迭代器失效，其余不受影响。
3. **map/set 等关联容器**
   仅**被删除节点**的迭代器失效，其他迭代器均有效。

### 迭代器的类别

迭代器类别	                                       核心能力	                                                                 典型示例
输入迭代器（Input）	                        只读、单向遍历（只能 ++），不可重复读	         std::istream_iterator
输出迭代器（Output）	                        只写、单向遍历（只能 ++），不可重复写	         std::ostream_iterator
前向迭代器（Forward）	                        可读可写、单向遍历，可重复读 / 写	                 std::forward_list迭代器
双向迭代器（Bidirectional）	                可读可写、双向遍历（++/--）	                                  std::list/std::map迭代器
随机访问迭代器（Random Access）	可读可写、任意跳转（±n、[]、比较），能力最强    std::vector/std::array迭代器、原生指针

### std::move的实现原理

一个类型转换工具，将左值强制转换为右值引用，让编译器认为这个对象“可以被移动”

### atomic变量的内存序

背景：编译器优化导致程序实际执行顺序跟代码顺序不完全一致

比如 x = 0 , y = 0

thread 1: x=1; y =1

thread 2 while(y!=1); assert(x == 1) // 可能会失败

以及现代计算机体系结构的特点：一个核赋值后在另一个核心中并不立马可见

### 如何防止死锁

导致死锁的4个必要条件：环形依赖（按固定顺序获取资源） ； 互斥（使用无锁编程）；  不可抢占（带超时的锁）；持有且等待（一次性获取资源；获取一个资源后立刻释放另一个）

## 算法

算法是操作容器中数据的通用函数模板，比如排序、查找、遍历、修改、数值计算等，几乎所有算法都依赖迭代器访问容器元素。
核心分类：
非修改型算法：for_each（遍历）、find（查找）、count（计数）
修改型算法：copy（复制）、replace（替换）、swap（交换）
排序算法：sort（快速排序）、stable_sort（稳定排序）、reverse（反转）
数值算法：accumulate（累加）、max_element（找最大值）

## 函数对象

也叫 “仿函数”，是 **重载了 ()` 运算符的类 / 结构体** ，可以像函数一样调用，常作为算法的参数（比如自定义排序规则）。

## 适配器

适配器是 STL 遵循 “适配器模式**” 设计的组件，核心作用是将一个已有组件（容器 / 迭代器 / 函数）的接口转换为另一种接口，以适配特定的使用场景

* 容器适配器：如 stack、queue、priority_queue
* 迭代器适配器：如 reverse_iterator、insert_iterator
* 函数适配器：如 bind、mem_fn

## 分配器

分配器是 STL 的底层内存管理工具，负责容器的内存分配与释放，默认使用std::allocator。用户也可以自定义分配器（比如实现内存池），但日常开发中几乎不需要修改默认分配器。

| 接口                  | 作用                                                                                               |
| --------------------- | -------------------------------------------------------------------------------------------------- |
| `allocate(n)`       | 分配 `n * sizeof(T)`字节的原始内存，返回 `T*`指针；⚠️ 内存未构造对象，不能直接访问           |
| `deallocate(p, n)`  | 释放 `p`指向的内存，`n`必须与 `allocate`的参数一致；⚠️ 释放前必须析构所有已构造的对象      |
| `construct(p, ...)` | 在 `p`指向的内存上构造 T 对象，参数为 T 的构造参数；底层调用 `new (p) T(...)`（placement new） |
| `destroy(p)`        | 析构 `p`指向的 T 对象，调用 T 的析构函数；底层调用 `p->~T()`                                   |
